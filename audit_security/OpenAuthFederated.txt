1. Access JWT verification does not pin the signing algorithm (alg / HS-RS confusion)  [CRITICAL]

SITUATION / PROBLEM:
Every call that verifies a library-minted token calls jose `jwtVerify` without an `algorithms` allowlist. The session cookie, the OAuth `state` cookie, the SAML relay cookie, the access tokens, and the public `verifyToken()`/`verifyMachineToken()` all verify with no pinned algorithm. jose will accept whatever algorithm the token header declares as long as the key type is compatible.

ROOT CAUSE:
`jwtVerify(token, key, opts)` is invoked with `opts` that never set `algorithms`. In embedded mode the key is a symmetric `Uint8Array` (HS256); in production it is a remote JWKS. With no allowlist, the verifier follows the token's own `alg` header rather than the one the issuer is supposed to use.

SCENARIO:
In production mode (`verify.ts`) tokens are expected to be RS256 verified against a JWKS. If an attacker submits a token whose header says `alg: HS256` and signs it using the PUBLIC RSA key bytes as an HMAC secret, a verifier that does not pin `algorithms: ['RS256']` can be tricked into HMAC-verifying with the public key (classic RS/HS confusion). Symmetrically, because no path pins HS256-only, a forged token presenting an unexpected algorithm header is evaluated on the attacker's terms.

RISK:
Full authentication bypass / token forgery, leading to arbitrary identity and arbitrary roles/permissions. Likelihood is high wherever a JWKS (asymmetric) path is used; even in embedded-only deployments the missing pin removes defense-in-depth against future algorithm-agility bugs.

REPRO STEPS:
1. Stand up the library in production mode (AUTH_EMBEDDED unset) with an RS256 JWKS issuer.
2. Take the issuer's public key, craft a JWT with header `{"alg":"HS256"}` and the desired claims, HMAC-sign it with the PEM public-key bytes.
3. Present it as a Bearer token to `verifyToken()` / `authenticateRequest()`.
4. Observe it verifies and yields attacker-chosen claims because no `algorithms` allowlist was set.

2. SAML assertion Audience is never validated against the SP Entity ID  [CRITICAL]

SITUATION / PROBLEM:
The SAML client is built passing `audience: cfg.spEntityId` to node-saml. In node-saml v5 the `audience` option configures the value placed into outbound AuthnRequests / metadata, NOT a check that the inbound assertion's `<AudienceRestriction><Audience>` equals the SP Entity ID. There is no explicit AudienceRestriction validation in the ACS path. An assertion minted for a DIFFERENT Service Provider can therefore be replayed into this SP.

ROOT CAUSE:
`buildSamlClient()` relies on a misread of the node-saml option; `validateSamlAcs()` consumes the resulting profile without independently asserting the assertion audience matches `spEntityId`.

SCENARIO:
The same IdP (e.g. a Google Workspace tenant) fronts several SAML SPs. A user signs into a low-value SP and the attacker captures that signed assertion (or an insider operates a malicious SP). Because this SP never checks that the assertion's audience is its own Entity ID, the captured assertion validates here and establishes a session as that user.

RISK:
Cross-SP assertion reuse → authentication as another user / privilege transfer between applications. High impact, plausible in any multi-SP IdP deployment.

REPRO STEPS:
1. Configure two SPs against one IdP; capture a valid signed SAML Response issued to SP-A.
2. POST that Response to SP-B's `/saml/acs` (this library) with a matching RelayState cookie crafted via the login route.
3. Observe a session is established despite the audience naming SP-A.

3. SAML response replay: InResponseTo disabled and no assertion-ID replay cache  [CRITICAL]

SITUATION / PROBLEM:
The SAML client sets `validateInResponseTo: ValidateInResponseTo.never`, so the binding between the AuthnRequest and the Response is not checked, and there is no cache of consumed assertion IDs or NotOnOrAfter-bounded one-time use. A signed SAML Response, once observed, can be replayed within its validity window (and the validity window is only loosely bounded — see clock-skew default).

ROOT CAUSE:
`buildSamlClient()` deliberately turns off InResponseTo validation ("would need cross-process shared storage"), substituting only a per-login RelayState cookie. RelayState is an unauthenticated, attacker-replayable value, not an assertion-replay defense.

SCENARIO:
An attacker who captures a victim's signed SAML Response (network logger, proxy, browser history of the POST body, or a logged ACS payload) initiates their own SAML login to obtain a fresh signed RelayState cookie, then POSTs the captured victim Response together with the matching RelayState. With no InResponseTo and no assertion-ID replay cache, the Response validates again and a session is minted for the victim.

RISK:
Session impersonation by SAML Response replay. High impact; feasible whenever a Response can be observed once.

REPRO STEPS:
1. Capture a valid SAML Response delivered to `/saml/acs`.
2. Hit `/saml/login` to mint a fresh `*_saml_relay` cookie and read back the RelayState it expects.
3. POST the captured SAMLResponse with that RelayState to `/saml/acs`.
4. Observe a session cookie is issued for the victim.

/Users/bryan/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/saml.ts (lines 76-89, 119-162)
/Users/bryan/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/frontend.ts (lines 1241-1296)

CALL COUNT:
2 call sites: handleSamlLogin and handleSamlAcs in frontend.ts (both route into saml.ts). Library-internal.

FIX:
* Root cause being fixed: no per-response one-time-use enforcement.
* How: enforce InResponseTo binding and/or a replay cache of consumed assertion IDs keyed until NotOnOrAfter.
* Structure of the fix: add a pluggable replay store (reuse the SessionStore abstraction or a small TTL cache); record each consumed assertion ID and reject duplicates; enable node-saml InResponseTo validation backed by that store, or bind the AuthnRequest ID into the signed relay cookie and require the Response's InResponseTo to equal it.
* Files changed and how: saml.ts — set `validateInResponseTo` to a real mode and surface/return the assertion ID and InResponseTo. frontend.ts — store the AuthnRequest ID in the signed relay cookie, compare it to the Response's InResponseTo at ACS, and reject already-seen assertion IDs.

SITUATION / PROBLEM:
When the embedding app does not supply `resolveGrants`, `defaultResolveGrants()` returns a membership with role `org:admin` and permissions `['*:read','*:write','org:sys_memberships:manage']` for EVERY authenticated identity. ROOT CAUSE:
The default grant function ignores the identity's groups entirely and hands out wildcard write plus membership-management to all comers; the library's permission model treats `*:write` and `org:sys_memberships:manage` as full authority.

SCENARIO:
An app is wired up with domain enforcement (good) but the developer forgets to pass `resolveGrants` (easy — it is optional). Privilege escalation to admin for the entire user base by default. High likelihood because the insecure behavior is the zero-config default and is silent.

2. Sign in any allowed-domain account.
3. Inspect the minted token / `<Protect permission="anything:write">` — it passes; `org:sys_memberships:manage` is granted.

5. Whole-SAML-Response signature not required (XML signature wrapping exposure)  [CRITICAL]

SITUATION / PROBLEM:
`wantAuthnResponseSigned` defaults to false; only the assertion is required to be signed. When the enclosing Response is unsigned, the surface for XML Signature Wrapping (XSW) attacks is larger: an attacker can wrap or relocate a legitimately-signed assertion inside a crafted unsigned Response so the parser reads attacker-chosen elements while the signature still "validates" against the original assertion.

ROOT CAUSE:
The config defaults the Response-level signature requirement off ("Google signs the assertion, not always the response") and depends entirely on the underlying library's XSW hardening, which is left at defaults and not explicitly asserted by this code.

SCENARIO:
An attacker obtains one validly-signed assertion (their own or captured), then constructs a Response document containing both the signed assertion and an injected assertion/attributes positioned so the signature check passes on the original while profile extraction reads the injected NameID/email. Because the Response is not required to be signed and audience/replay checks are weak (findings 2 and 3), the forged identity is accepted.

RISK:
Identity forgery / authentication bypass via signature wrapping. High impact; exploitability depends on the parser but the missing Response-signature requirement removes a key mitigation.

REPRO STEPS:
1. Acquire one valid signed assertion from the IdP.
2. Build a Response wrapping it with an injected assertion carrying an attacker email.
3. POST to `/saml/acs` with a valid RelayState cookie.
4. Observe whether the injected identity is accepted (depends on XSW robustness, which this code does not enforce).

8. Cookies default to non-Secure and SameSite=Lax (CSRF / cookie theft over HTTP)  [HIGH]

SITUATION / PROBLEM:
`cookieSecure` defaults to false and all cookies are written with `SameSite=Lax` and only conditionally `Secure`. The session cookie, OAuth state cookie, and SAML relay cookie can therefore be transmitted over plain HTTP and are not `SameSite=Strict`. The SAML ACS is a cross-site top-level POST, which `SameSite=Lax` does NOT send cookies for in modern browsers (functional break) while still leaving the relay cookie exposed if `Secure` is off.

ROOT CAUSE:
`normalizeConfig` defaults `cookieSecure` to false; `setCookie`/`clearCookie` hardcode `SameSite=Lax` and add `Secure` only when the flag is set.

SCENARIO:
The library is deployed without explicitly setting `cookieSecure: true` (the insecure default). A network attacker on the same LAN/Wi-Fi reads the session cookie over HTTP and replays it. Because cookies are not port-isolated and default to a shared prefix, a sibling app on the host can also read them.

RISK:
Session hijacking over HTTP; CSRF surface from Lax defaults. High likelihood given the unsafe default.

REPRO STEPS:
1. Deploy with defaults (no cookieSecure).
2. Sign in over HTTP; capture the `oaf_session` cookie on the wire.
3. Replay it to access the session.

13. Session-store self-heal re-creates a session from a cookie, defeating revocation  [HIGH]

SITUATION / PROBLEM:
`readSession()` has a "self-heal" path: when a valid signed cookie presents a `sid` for which the store has NO record, it re-creates the durable record from the cookie (`revoked: false`). Because `remove()` tombstones by keeping a `revoked` record, revocation survives — but if the tombstone file is ever lost, cleared, rotated, or the store is swapped/emptied, a still-valid long-lived cookie silently re-creates an active session. ROOT CAUSE:
The self-heal trusts the signed cookie as sufficient to mint a fresh non-revoked record whenever no record is found, conflating "no record" (which can mean "tombstone was lost") with "first request after upgrade".

SCENARIO:
An admin revokes a user (tombstone written). Later the store directory is cleaned/migrated, or a different store instance is used. The user replays their old (still in-date) session cookie; readSession finds no record and re-creates an active one. Revocation is undone.

RISK:
Revocation bypass / session resurrection, especially severe with multi-year cookie lifetimes. Medium likelihood, high impact (offboarding is a core promise).

REPRO STEPS:
1. Sign in (record created), then revoke (tombstone).
2. Delete/clear the session store directory (simulating rotation/migration).
3. Replay the original session cookie; observe a new active record is created and access is granted.
