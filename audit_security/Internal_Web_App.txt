1. Hardcoded shared HS256 secret lets anyone forge session/access tokens  [CRITICAL]

SITUATION / PROBLEM:
The string "local-dev-shared-secret-change-me" is committed in the frontend source and is also the backend's default value for AUTH_SESSION_SECRET. In embedded mode the SAME HS256 secret signs and verifies BOTH the session cookie and the short-lived access tokens. Anyone who reads the repository knows the secret and can mint a valid access token for any user/roles/permissions.

ROOT CAUSE:
A "dev convenience" shared secret was baked into source (frontend resolveDevAuth.ts) and wired as the backend fallback (auth-frontend.ts) so the dev mock's mint-secret equals the backend's verify-secret. The library's embedded verifier only rejects the one well-known literal "dev-shared-secret"; this app deliberately chose a DIFFERENT literal precisely to slip past that guard, so the library never refuses it.

SCENARIO:
An attacker clones the repo (or reads the frontend bundle), forges an HS256 JWT with alg HS256, sub=user_x, roles=["org:admin"], permissions=["*:write","org:sys_memberships:manage"], iss matching the configured issuer, and a 1-minute exp signed with the committed secret. They send it as a Bearer token to any protected route. AuthFederatedStrategy verifies it against the same secret and authorizes the request with admin permissions.

RISK:
Full authentication and authorization bypass. An attacker becomes any employee with any role/permission. High impact; trivial likelihood if the secret is not overridden in the deployment env.

REPRO STEPS:
1. Read LOCAL_DEV_SHARED_SECRET from the frontend source.
2. Sign an HS256 JWT carrying sub, email on an allowed domain, roles, permissions, and a short exp.
3. Call GET /api/auth/me (and any RBAC-gated route) with Authorization: Bearer <forged>.
4. Observe a 200 with the forged identity / permissions.

2. Backend forces AUTH_EMBEDDED=true and a fallback secret at boot, even in production  [CRITICAL]

SITUATION / PROBLEM:
buildAuthFrontend() sets process.env.AUTH_EMBEDDED = "true" and process.env.AUTH_SESSION_SECRET to a resolved value whenever those are unset, at every boot, with no production guard. This pins the whole app into embedded HS256 mode (symmetric secret can both mint and verify) instead of asymmetric JWKS verification, and guarantees a secret is always present — including the hardcoded fallback from Finding 1.

ROOT CAUSE:
Bootstrap-time mutation of process.env to make mint-secret === verify-secret "work out of the box". There is no branch that, in production, demands an explicit strong secret or switches to JWKS/RS256.

SCENARIO:
The app is deployed with neither AUTH_SESSION_SECRET nor AUTH_EMBEDDED set. At boot the code sets AUTH_EMBEDDED=true and AUTH_SESSION_SECRET=local-dev-shared-secret-change-me. The production app now verifies forged HS256 tokens (Finding 1) and never reaches the safer JWKS path.

RISK:
Production silently runs in symmetric-secret embedded mode with a known secret. Combined with Finding 1 this is a total auth bypass. High impact; medium likelihood (depends on deploy hygiene, but the code makes the insecure path the default).

REPRO STEPS:
1. Start the backend with NODE_ENV=production and no AUTH_SESSION_SECRET / AUTH_EMBEDDED.
2. Inspect process.env after bootstrap — both are now set to the embedded defaults.
3. Forge a token per Finding 1 and authenticate.

3. Server-side session revocation / offboarding silently does nothing (wrong store)  [CRITICAL]

SITUATION / PROBLEM:
The app's sign-out and admin "revoke all sessions" call federatedClient.sessions.* which performs HTTP requests to AUTH_BACKEND_API (default https://api.localhost/v1). But this deployment runs embedded with a FileSessionStore on disk; there is no Backend API server. The revoke/list/verify calls therefore fail (or hit a non-existent host) and the durable embedded session record is never tombstoned. The "immediate revocation despite short token life" guarantee the app advertises does not hold.

ROOT CAUSE:
Two parallel session models are mixed: the embedded createFederatedFrontend + FileSessionStore (the real store for this app) versus the remote AuthClient REST resources. SessionsService was wired to the REST resources, not to the embedded FileSessionStore actually backing sessions.

SCENARIO:
An employee is offboarded. An admin calls POST /api/sessions/revoke-user. SessionsService.getSessionList hits https://api.localhost/v1/sessions and throws; the catch logs a warning and returns 0. The offboarded user's on-disk session record stays valid; because the session cookie self-heals and TTL is ~10 years (Finding 6), the user keeps minting access tokens and retains access.

RISK:
Revocation, sign-out-everywhere, and offboarding are non-functional. A terminated or compromised user cannot be cut off. High impact; high likelihood (any revoke attempt).

REPRO STEPS:
1. Sign in (a sess_*.json appears under the FileSessionStore root).
2. Call POST /api/sessions/sign-out or /api/sessions/revoke-user.
3. Observe the call errors against api.localhost and the on-disk record is unchanged / not revoked.
4. Continue calling protected routes with the still-valid cookie/token.

4. SAML accepts assertion without requiring the Response signature, with replay protection disabled  [HIGH]

SITUATION / PROBLEM:
When SAML is enabled, this app passes config straight through to the library. The library defaults wantAuthnResponseSigned to false (only the assertion is required signed) AND sets validateInResponseTo to "never". The app sets neither, so the SP accepts a SAML Response whose outer envelope is unsigned and performs no InResponseTo binding between the AuthnRequest and the Response. The only anti-replay control is the app-level RelayState cookie.

ROOT CAUSE:
Library defaults (saml.ts) chosen for Google compatibility, combined with the app never overriding wantAuthnResponseSigned, and InResponseTo validation being turned off in favor of a RelayState cookie. An attacker-replayed or wrapped Response is harder to reject without response-level signing + InResponseTo.

SCENARIO:
An attacker who captures a victim's signed assertion (from logs, a proxy, or a prior exchange) replays it to /api/v1/saml/acs. With InResponseTo not validated and the Response envelope unsigned, the SP relies only on the assertion's NotOnOrAfter window and the RelayState cookie; within that window, and if the attacker can supply a matching RelayState cookie (same-site or via a forged login start), the assertion can be replayed. XML signature-wrapping risk is also elevated without response-level signing.

RISK:
SAML assertion replay / signature-wrapping leading to session establishment as the victim. High impact; medium likelihood (requires assertion capture or wrapping).

REPRO STEPS:
1. Enable SAML (AUTH_SAML_ENABLED=true).
2. Capture a valid signed assertion within its validity window.
3. POST it to /api/v1/saml/acs with a self-consistent RelayState cookie/param.
4. Observe a session cookie issued for the victim.

5. SAML sign-in trusts email as verified unconditionally  [HIGH]

SITUATION / PROBLEM:
On the SAML path the library hardcodes emailVerified: true for the mapped identity, then the shared finishSignIn enforces domain only on that already-trusted email. There is no equivalent of the OIDC email_verified check. If an IdP (or a mis-mapped attribute / a non-Google IdP this SP could be pointed at) emits an email the domain of which is allowed but which is not actually controlled/verified, the SP accepts it.

ROOT CAUSE:
saml.ts assumes "a valid signed Workspace assertion implies a verified email" and sets emailVerified true regardless of IdP attributes; the app does not constrain idpEntityId/issuer to Google, so the assumption can be violated.

SCENARIO:
The SP is configured against an IdP that lets a user set an arbitrary email attribute on an allowed domain (or NameID is spoofable). The signed assertion passes; emailVerified is forced true; domain enforcement passes; the attacker is signed in as someone@alloweddomain.

RISK:
Account takeover / impersonation within an allowed domain. High impact; likelihood depends on IdP, but the code removes a control that should exist.

REPRO STEPS:
1. Point the SP at an IdP that does not strictly verify the email attribute.
2. Authenticate with an email on an allowed domain you do not own.
3. Observe a session established for that email.

6. Effectively permanent sessions: ~10-year TTL, no idle timeout, no sid rotation  [HIGH]

SITUATION / PROBLEM:
The app sets sessionTtlSeconds to ~10 years and inactivityTimeoutSeconds to 0. The session cookie is a sliding window re-issued on each token mint, and the session id is never rotated on sign-in or privilege change. So a stolen session cookie is usable for a decade, no idle timeout ever closes it, and there is no fixation/rotation defense.

ROOT CAUSE:
Deliberate "stay signed in forever" configuration in auth-frontend.ts (lines 88-100) plus a library that never rotates sid (frontend.ts mints/refreshes but keeps the same sid; reverify only stamps lastVerifiedAt).

SCENARIO:
An attacker steals the session cookie (XSS-adjacent, shared machine, backup of the disk store, or a logged value). Because there is no idle timeout, no rotation, and a 10-year absolute TTL, the cookie keeps minting access tokens indefinitely; even sign-out is unreliable (Finding 3).

RISK:
Long-lived session hijack with no natural expiry and no rotation. High impact; medium likelihood.

REPRO STEPS:
1. Sign in; capture the session cookie.
2. Wait arbitrarily long without activity.
3. Replay the cookie to /api/v1/client/sessions/:id/tokens and obtain a fresh access token.

SOURCE FILES INVOLVED:
  /Users/bryan/BGit/all/app/code/packages/backend/src/modules/auth/auth-frontend.ts (lines 88-130)
  /Users/bryan/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/frontend.ts (lines 515-525, 1003-1044, 1059-1139)

CALL COUNT:
1 app call site configures TTL/inactivity (auth-frontend.ts); the library mint/touch/reverify handlers reuse the same sid.

FIX:
* Root cause being fixed: no idle timeout, decade-long absolute TTL, no sid rotation.
* How: set a sane inactivity timeout and a shorter maximum lifetime; rotate sid on sign-in and on privilege/org change.

7. State-changing session endpoints are cookie-authenticated with no CSRF defense  [HIGH]

SITUATION / PROBLEM:
The embedded Frontend API mounted at /api/v1 authenticates the session via the oaf_act3hub_session cookie (SameSite=Lax). Several state-changing endpoints — POST /tokens (mint), POST /touch (set active org), POST /remove (sign out), GET /reverify — act purely on the cookie with no CSRF token / double-submit / Origin check. SameSite=Lax does not protect top-level POST navigations and does not protect GET /reverify.

ROOT CAUSE:
The library relies on SameSite alone and uses the Authorization: Bearer publishableKey header (not a secret) as the only non-cookie input. No CSRF token is required for cookie-authenticated mutations.

SCENARIO:
A logged-in employee visits a malicious page that auto-submits a form POST to /api/v1/client/sessions/<id>/remove (or triggers GET /reverify). The browser attaches the session cookie; the action executes cross-site (sign the user out, or flip active org / RBAC scope mid-session).

RISK:
CSRF on session lifecycle and active-org/RBAC scope. Medium-high impact; medium likelihood.

REPRO STEPS:
1. While signed in, load an attacker page that issues a cross-site POST to /api/v1/client/sessions/<known-sid>/remove or /touch.
2. Observe the session changes without the user's intent.
