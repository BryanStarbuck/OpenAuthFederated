1. Almost the entire /v1 data plane is mounted with no auth middleware  [CRITICAL]

SITUATION / PROBLEM:
The app builds a real auth layer (requireAuth / requireAdmin / optionalAuth) but
wires it into only a small set of routes. 67 of 76 route files under
src/routes register their handlers with NO auth middleware and never read
req.user / req.claims. There is no app-level app.use(requireAuth) and no router
that blanket-guards /v1. Entity CRUD, the AI engine, publish, knowledge,
content store, assets, destinations, create-post / create-reply, read-data,
sync, and more are reachable by any anonymous caller who can hit the port.

ROOT CAUSE:
Auth was implemented as opt-in per-route middleware, and the 8-agent
parallel-build pattern (registerAgent1..8 plus dozens of feature routers) added
hundreds of handlers without attaching the guard. server.ts mounts the library
Frontend API and the compat auth routes but never installs a default guard for
the rest of /v1.

SCENARIO:
An attacker (or any unauthenticated browser/script) reaches the server on
:7700 (or the deployed origin) and calls POST /v1/publish/:postId,
POST /v1/ai/proposals/post, GET /v1/knowledge, POST /v1/create-post/draft,
the entity CRUD endpoints, etc. None of them check identity, so the attacker
reads and mutates all marketing data and triggers AI spend and cross-network
publishing.

RISK:
Full read/write of application data and side-effecting actions (publish, AI
generation, data export, deletion) with zero authentication. This is the single
largest exposure in the app.

REPRO STEPS:
1. Start the app (DEPLOY_MODE=localhost) with no credentials presented.
2. curl http://localhost:7700/v1/knowledge  -> 200 with data, no 401.
3. curl -X POST http://localhost:7700/v1/create-post/draft -d '{...}' -> handled.
4. Observe no Authorization header / cookie was required.

2. Data export and account-deletion endpoints are unauthenticated with client-supplied tenantId (cross-tenant export/delete)  [CRITICAL]

SITUATION / PROBLEM:
The GDPR-grade data export and account-deletion endpoints take tenantId from the
request body/query and perform the action for that tenant, with no auth
middleware and no binding to the caller's verified identity. The code comments
even call its tenantId check an "IDOR guard," but the tenantId is supplied by
the attacker.

ROOT CAUSE:
The route trusts req.body.tenantId / req.query.tenantId as authorization rather
than deriving the tenant from a verified session, and the file is in the
unguarded majority (no requireAuth).

SCENARIO:
An attacker enumerates or guesses tenant ids ("act3", a customer slug) and:
- POST /v1/data/export {tenantId} to dump another tenant's entire dataset, then
- GET /v1/data/export/:job_id/archive?tenantId=... to download it, or
- POST /v1/data/delete-account {tenantId, requestedBy} to schedule destruction
  of another tenant's data.

RISK:
Mass data exfiltration of any tenant and destructive deletion of any tenant.
Critical confidentiality and availability impact.

REPRO STEPS:
1. curl -X POST http://localhost:7700/v1/data/export -H 'Content-Type: application/json' -d '{"tenantId":"act3"}'
2. Note the returned job id.
3. curl 'http://localhost:7700/v1/data/export/<job>/archive?tenantId=act3' -> archive.
4. curl -X POST .../v1/data/delete-account -d '{"tenantId":"act3","requestedBy":"x"}'.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/data_export.ts (lines 42-160; tenantId from body/query, no requireAuth)
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/index.ts (registerDataExport, no guard)
- /Users/bryan/BGit/all/marketing/ai/code/src/middleware/auth.ts (the unused guard)

CALL COUNT:
6+ unauthenticated handlers in one file (export start/list/status/archive,
delete-account start/list/cancel), all in src/routes/data_export.ts.

FIX:
- Root cause being fixed: authorization based on attacker-supplied tenantId.
- How: require authentication and derive tenant strictly from req.user.tenant_id;
  reject any body/query tenantId that does not equal the verified tenant.
- Structure of the fix: add requireAuth (and requireAdmin for delete-account) to
  every data_export route; replace tenantFromBody/query with req.user.tenant_id.
- Files changed and how:
  * src/routes/data_export.ts: add guards; source tenantId from req.user; drop
    the body/query tenantId path or use it only to 403 on mismatch.
  * src/server.ts: covered by the global /v1 guard from finding 1.

3. Settings tenant-write defaults to allow when no auth user is present (default-allow admin)  [CRITICAL]

SITUATION / PROBLEM:
PUT /v1/settings/tenant is protected by an isAdminReq() that returns true when
there is no req.user at all (its comment: "when there is no auth user ... we
allow tenant writes"). Because the settings routes are mounted without
requireAuth (finding 1), req.user is never populated, so isAdminReq() always
returns true and ANY anonymous caller can write tenant-wide settings.

ROOT CAUSE:
Fail-open authorization: absence of an identity is treated as "trusted local
operator" instead of "deny." Combined with the missing guard, the "localhost
convenience" branch is the live production branch.

SCENARIO:
An unauthenticated attacker calls PUT /v1/settings/tenant?tenantId=act3 with a
malicious settings patch and changes tenant-level configuration for everyone.

RISK:
Unauthenticated tenant-wide configuration tampering. Critical integrity impact;
can be used to pivot (e.g. flip flags, change limits).

REPRO STEPS:
1. curl -X PUT 'http://localhost:7700/v1/settings/tenant?tenantId=act3' -H 'Content-Type: application/json' -d '{"someSetting":"attacker"}'
2. Observe 200 (not 403) and the tenant settings change.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/settings.ts (lines 69-78 isAdminReq default-true; lines 141-155 PUT tenant)
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/index.ts (registerSettings, no guard)

CALL COUNT:
1 default-allow authorization function gating 1 tenant-write route (plus the
user routes in the same file). All in src/routes/settings.ts.

FIX:
- Root cause being fixed: fail-open admin check.
- How: deny by default; require a verified admin identity.
- Structure of the fix: mount requireAuth/requireAdmin on the settings routes and
  make isAdminReq() return false (not true) when req.user is absent.
- Files changed and how:
  * src/routes/settings.ts: invert the missing-user branch to deny; require
    req.user.is_admin === true for tenant writes.
  * src/server.ts / index.ts: apply the global /v1 guard.

5. CORS reflects any Origin with credentials:true (cross-site authenticated requests)  [CRITICAL]

SITUATION / PROBLEM:
server.ts uses cors({ origin: true, credentials: true }). origin:true echoes the
caller's Origin into Access-Control-Allow-Origin and allows credentials, so any
website the victim visits can make credentialed cross-site requests (the
oaf_mai_session cookie rides along) and read the responses.

ROOT CAUSE:
Reflecting the request Origin instead of an allowlist, while also allowing
credentials, is the canonical CORS misconfiguration.

SCENARIO:
A signed-in operator visits evil.com. evil.com's JavaScript does
fetch('http://localhost:7700/v1/...', { credentials:'include' }); the browser
sends the session cookie, CORS reflects evil.com, and the script reads the
response. Combined with findings 1-4 this means full cross-origin data theft.

RISK:
Cross-site read of authenticated responses and (with the session cookie) any
state-changing call that does not require a custom header. Critical.

REPRO STEPS:
1. From a page on a different origin, fetch a /v1 endpoint with credentials:'include'.
2. Observe Access-Control-Allow-Origin echoes the page's origin and the response
   is readable.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/server.ts (line 228, cors config)

CALL COUNT:
1 CORS configuration call in src/server.ts (line 228) affecting every route.

FIX:
- Root cause being fixed: Origin reflection with credentials.
- How: restrict origin to an explicit allowlist (PUBLIC_ORIGIN + known app
  origins); keep credentials only for those origins.
- Structure of the fix: replace origin:true with a function/array allowlist.
- Files changed and how:
  * src/server.ts: cors({ origin: [PUBLIC_ORIGIN, ...allowed], credentials: true }).

6. Degraded no-DB mode trusts the token's own sub/email to synthesize an admin identity  [HIGH]

SITUATION / PROBLEM:
resolveUser(), when upsertFromGoogle() throws (e.g. no Postgres), synthesizes a
UserRow purely from the token claims, setting is_admin from the hardcoded email
allowlist. The id/email/google_sub all come from the (valid) token, but the
identity is no longer backed by any server-side record — admin status hinges only
on the email claim matching the allowlist.

ROOT CAUSE:
A fail-open degraded path that mints a working (and possibly admin) identity from
claims when the database is unavailable, instead of failing closed.

SCENARIO:
If the DB is down (or made to appear down) and an attacker holds any valid
allowlisted-domain token, the degraded branch grants them a session; if the email
is one of the three hardcoded admin emails, full admin — with no DB row, audit, or
revocation backing it.

RISK:
Authentication/authorization that survives loss of the source-of-truth store, and
admin elevation tied solely to an email-claim allowlist. High.

REPRO STEPS:
1. Stop Postgres (or induce upsertFromGoogle to throw).
2. Present a valid allowed-domain token to any guarded route.
3. Observe the request is accepted; with an allowlisted email, is_admin=true.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/middleware/auth.ts (lines 161-189 degraded synthesis; line 175 admin from allowlist)
- /Users/bryan/BGit/all/marketing/ai/code/src/lib/admin_allowlist.ts (lines 13-22)

1 degraded-mode branch in src/middleware/auth.ts, reached by every resolveUser

7. Session/auth cookies are not Secure in localhost mode and SameSite=Lax allows top-level CSRF on auth routes  [HIGH]

SITUATION / PROBLEM:
The library sets the session, OAuth state, and SAML relay cookies with
cookieSecure driven by IS_SERVER, so in localhost/dev they lack Secure. All
cookies are SameSite=Lax (library default), which still sends them on top-level
GET navigations — and the app exposes state-changing GETs like
GET /v1/auth/logout. The app also passes cookieSecure: IS_SERVER, so any non-https
deployment (or a proxy that terminates TLS upstream without setting IS_SERVER)
ships non-Secure session cookies.

ROOT CAUSE:
Secure flag is tied to a deploy-mode boolean rather than the actual transport, and
SameSite=Lax combined with GET side effects leaves a CSRF window.

SCENARIO:
On a mixed/non-https path the session cookie can leak over plaintext; and a
cross-site <img src=".../v1/auth/logout"> or top-level link logs the victim out
(and other Lax-sendable GET side effects fire).

RISK:
Session cookie disclosure on non-TLS transport and CSRF on GET-side-effect auth
routes. High.

REPRO STEPS:
1. Run with DEPLOY_MODE=localhost behind any non-localhost reachable interface.
2. Inspect Set-Cookie for oaf_mai_session — no Secure attribute.
3. From another origin, navigate to GET /v1/auth/logout and observe logout.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/lib/auth_frontend.ts (line 85 cookieSecure: IS_SERVER)
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/auth.ts (lines 197-208 GET/POST logout; clearCookie)
- /Users/bryan/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/frontend.ts (lines 252-274 cookie attributes; SameSite=Lax default)

CALL COUNT:
1 cookieSecure setting (auth_frontend.ts line 85) plus the library cookie helpers;
1 GET logout route (auth.ts line 204).

FIX:
- Root cause being fixed: Secure flag not bound to real transport; CSRF-able GETs.
- How: set cookieSecure from the actual scheme (PUBLIC_ORIGIN https / X-Forwarded-Proto),

The app authenticates from a cookie without any CSRF defense, relying only on
SameSite=Lax (which is insufficient for top-level navigations and some
sub-request shapes, and is non-Secure in dev per finding 7).

A logged-in operator visits a malicious page that auto-submits a form / fetch to a
POST /v1/... route; the browser attaches the session cookie and the action runs
under the victim's identity.

RISK:
Cross-site request forgery on all cookie-authenticated mutating endpoints. High.

REPRO STEPS:
1. Sign in (session cookie set).
2. From another origin, submit a POST to a guarded mutating route with
   credentials included and no Authorization header.
3. Observe the request authenticates via the cookie.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/middleware/auth.ts (lines 92-101 cookie token acceptance)
- /Users/bryan/BGit/all/marketing/ai/code/src/server.ts (no CSRF middleware installed)

1 token extractor (auth.ts lines 92-101) feeding every guard call site (24+

9. HMAC ingest verifies over re-serialized JSON because no rawBody is captured (signature bypass / canonicalization gap)  [HIGH]

SITUATION / PROBLEM:
The cloud ingest endpoint authenticates with HMAC-SHA256 over
`${timestamp}\n${rawBody}`. rawBodyFor() reads req.rawBody, but server.ts mounts
express.json() WITHOUT a verify hook, so req.rawBody is never populated. The code
falls into its "degraded" branch and HMACs JSON.stringify(req.body) — a
re-serialized form that does not equal the producer's signed bytes (key order,
whitespace, unicode escaping differ), so either legitimate requests fail or, if
the producer also signs re-serialized output, the canonicalization is attacker-
influenceable.

ROOT CAUSE:
The raw request bytes the signature is computed over are not preserved end to end;
the verifier reconstructs them instead of using the exact signed bytes.

SCENARIO:
An attacker exploits the canonicalization gap (e.g. duplicate/reordered keys,
differing escaping) so a body that re-serializes to the signed string is accepted
while carrying different effective content, or downgrades verification to the weak
JSON.stringify path.

RISK:
HMAC authentication that does not bind to the actual transmitted bytes —
signature confusion / potential bypass on the ingest channel. High (ingest writes
into the knowledge base consumed by the AI engine).

REPRO STEPS:
1. Send POST /v1/ingest/extension with valid headers.
2. Observe req.rawBody is undefined (no verify hook) and the verifier HMACs the
   re-serialized body, not the raw bytes.

SOURCE FILES INVOLVED:
- /Users/bryan/BGit/all/marketing/ai/code/src/routes/ingest.ts (lines 84-95 rawBodyFor; lines 173-189 HMAC)
- /Users/bryan/BGit/all/marketing/ai/code/src/server.ts (line 235 express.json with no verify hook)

CALL COUNT:
1 HMAC verification path in src/routes/ingest.ts; depends on the single
express.json() config in src/server.ts line 235.

FIX:
- Root cause being fixed: signature not bound to raw bytes.
- How: capture the raw body via an express.json({ verify }) hook and HMAC exactly
  those bytes; remove the JSON.stringify fallback (fail closed if rawBody absent).
- Files changed and how:
  * src/server.ts: add verify:(req,_res,buf)=>{req.rawBody=buf} to express.json.
  * src/routes/ingest.ts: if rawBody is missing, reject instead of re-serializing.

SITUATION / PROBLEM:
SESSION_TTL_SECONDS defaults to 10 years and AUTH_INACTIVITY_TIMEOUT_SECONDS to 0

shared machine remains valid effectively forever; the operator has no automatic
idle logout.

RISK:
Long-lived credential theft window and no idle-timeout backstop. High.

REPRO STEPS:
1. Sign in; capture the oaf_mai_session cookie.
2. Replay it weeks later without any activity -> still authenticated.

11. AUTH_SESSION_SECRET falls back to the legacy weak JWT_SECRET constant; forgeable tokens  [HIGH]

SITUATION / PROBLEM:
AUTH_SESSION_SECRET resolves to AUTH_SESSION_SECRET env, then JWT_SECRET env,
then (localhost) a disk-persisted random secret, then (server) the JWT_SECRET
constant 'dev-only-change-me-mai-localhost-secret'. In server mode validateConfig
fatals only if BOTH AUTH_SESSION_SECRET and JWT_SECRET env are unset; if an
operator set JWT_SECRET to the well-known dev string (or a weak value), the
